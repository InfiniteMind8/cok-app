# Prompt F.2 — PWA and Play Store packaging

**Block:** F — Production readiness
**Status:** Done (conditional) (2026-05-09)

## Brief
Final Phase 1+ build prompt. Turn the existing Next.js 16 production build into an installable PWA, scaffold a Trusted Web Activity (TWA) wrap so a signed Android App Bundle (AAB) can be produced, and prepare every Play Console listing asset and submission runbook the project owner needs to ship the app — web first, then mobile.

## Acceptance Criteria
1. Deployed app passes Chrome's PWA installability audit (manifest + service worker present and valid).
2. A signed AAB exists at a known path and can be installed on a test Android device, opens full-screen, and the app functions identically.
3. All Play Console listing assets exist and are brand-correct.
4. `docs/play-store.md` is a complete submission runbook.

## Implementation
- **Manifest** (`website/app/manifest.ts`): `start_url='/'`, `scope='/'`, `lang='en'`, full icon set (192/256/384/512 plus 512 maskable), `categories: ['lifestyle','finance','productivity']`. Brand colours unchanged.
- **Icons** (`website/public/icons/`): five PNGs generated by `scripts/generate-pwa-icons.ts` (sharp). Maskable variant uses 70% safe-area on `karis-stone-50` background. Same script writes the high-detail Play Store listing icon to `marketing/play-store/icon-512.png`.
- **Service worker** (`website/public/sw.js`): hand-rolled (~120 lines, no `next-pwa` / `@serwist/next` to avoid Turbopack + `withSentryConfig` integration risk per `website/AGENTS.md` "this is not the Next.js you know"). Caching: cache-first for `/_next/static/`, `/icons/`, fonts; network-first with `/offline` fallback for navigations; stale-while-revalidate for image responses. Auth/API/webhook/cron paths bypassed entirely. `VERSION` constant at the top — bump per release. Decision: `D-F2-01`.
- **SW registration** (`website/components/shared/sw-register.tsx`): client-only effect, registers `/sw.js` only when `process.env.NODE_ENV === 'production'` and `'serviceWorker' in navigator`.
- **Offline fallback** (`website/app/offline/page.tsx`): brand-styled, no auth, no DB. Targeted by the SW navigation fallback.
- **Install prompt** (`website/components/shared/install-prompt.tsx`): handles `beforeinstallprompt` for Chromium / Android. Banner with Install + Dismiss; mirrors the existing `IOSInstallPrompt`. Both prompts mounted in `app/layout.tsx`; UA-keyed exclusivity at runtime.
- **Legal routes**: `app/privacy/page.tsx` + `app/terms/page.tsx` are server components that read `legal/privacy.md` / `legal/terms.md` at build time and render via a hand-rolled `LegalDocument` component (`components/legal/`, no new markdown dep). Drafts open with a "DRAFT — pending counsel review" banner. Decision: `D-F2-02`.
- **TWA scaffolding**: `marketing/play-store/twa-manifest.json` (Bubblewrap-compatible, `org.cityofkaris.app`, host placeholder), and `website/public/.well-known/assetlinks.json` (placeholder SHA-256 fingerprint, owner replaces post-`bubblewrap build`). Decision: `D-F2-03`.
- **Listing assets**:
  - `marketing/play-store/icon-512.png` — generated alongside PWA icons.
  - `marketing/play-store/feature-graphic.png` — 1024×500, generated by `scripts/generate-feature-graphic.ts` (sharp + SVG overlay; no font file dep).
  - `marketing/play-store/screenshots/` — generated by `scripts/capture-play-store-screenshots.ts` (Playwright at 1080×1920 against a local production build via the D.10 demo-mode auth path).
- **Listing copy**: `marketing/play-store/listing-copy.md` — short description verbatim per playbook; full description ≤4000 chars, brand-voice, covers all four roles; explicitly notes the app is invitation-only.
- **Data safety**: `website/docs/play-store-data-safety.md` — pre-filled answers for transcription into Play Console.
- **Submission runbook**: `website/docs/play-store.md` — Bubblewrap install + commands, key management section (never-commit, password manager + offline backup, rotation policy referencing R-D11-01 / D-D12-03), Asset Links update procedure, full Play Console step-by-step (developer account → app → listing → content rating → data safety → target audience → AAB upload → internal track → closed → open → production), pre-submission checklist, post-submission monitoring, common rejection reasons.
- **Master go-live runbook**: `website/docs/go-live.md` — chronological owner runbook, web-first → mobile, sequences existing setup docs + `production-deploy.md` + `play-store.md` end-to-end. Phases A–I covering account provisioning, deploy, web verification, PWA verification, counsel review, mobile build, Play Console submission, stabilisation, post-launch monitoring. Single master pre-go-live checklist at the bottom.

## Out of scope
- Running `bubblewrap init` / `bubblewrap build` (owner — needs JDK + Android signing keystore on a workstation outside the build agent's environment).
- Installing the AAB on a real Android device (owner).
- Submitting to Play Console / paying the $25 fee (owner — playbook §F.2 step 5 explicit).
- Counsel review of `legal/privacy.md` and `legal/terms.md` (owner + counsel).
- Performing the actual account setups in Resend / Clerk / Upstash / S3 / Supabase / Sentry / Play Console (owner — `docs/go-live.md` documents every step but only the owner can sign in to those provider dashboards).
- Block F closure checkpoint and §9 final readiness audit (separate engagement after F.2).
